# Write a PRD

## 概述

将模糊的产品意图、零散的需求输入，整理成符合部门规范的完整 PRD 文档。
严格遵循 `references/prd-template.md` 中定义的章节结构、表格格式和内容规范，产出可直接用于评审和交付的 Markdown PRD。

## 工作流程

### 1. 理解输入，识别产品类型

- 从混杂输入中提取核心信息：产品目标、痛点、功能点、用户角色、约束条件。
- 判断产品类型：**数字化产品** 还是 **AI 产品**（影响"功能详细说明"章节的内容要求）。
- 识别已知事实与合理假设，区分"本期必须有"和"可选延伸项"。
- 在能帮助理解业务语义时，保留用户原有术语。

### 2. 主动补齐缺失信息

- 如果合理假设可以推动产出，不要因为"信息不完整"就停止。
- 对缺失字段补充合理默认值，例如：
  - 版本号默认 V1.0，日期使用当前日期，负责人填写"待定"
  - 优先级根据功能重要性推断（P0/P1/P2）
  - 非功能需求中的性能指标参考行业通用标准
- 将所有假设和待确认项统一记录在文档末尾的"待确认问题"小节。
- 只有在某个选择会显著改变产品方向时，才需要向用户升级确认。

### 3. 按规范结构组织文档

严格按照 `references/prd-template.md` 定义的章节顺序和格式要求输出：

1. **版本信息**：表格形式，含版本号、创建日期、负责人
2. **变更日志**：表格形式，含时间、版本号、变更人、主要变更内容
3. **文档说明**：名词解释表格，对文档中出现的专业术语和系统简称进行解释
4. **需求背景**：需求说明（背景、痛点、目标）+ 竞品分析（可选）
5. **功能列表**：五列表格（功能名称、功能概述、适用角色、优先级、父记录）
6. **功能详细说明**：根据产品类型选择对应内容规范
7. **非功能需求**：性能指标表格 + 视觉/运营/法律/安全需求说明
8. **数据埋点**：埋点事件表格，含参数名、说明和值
9. **附录**：相关文档链接（数据分析报告、用户调研报告等）
10. **评审记录**：评审信息表格 + 问题记录
11. **待确认问题 / 默认假设**（如有）

### 4. 功能详细说明的差异化处理

#### 数字化产品
- 用户旅程：描述主要操作路径（文字描述，注明"建议使用画板工具绘制"）
- 交互原型图：注明"待设计师提供 Figma 链接"
- 原型功能说明：先描述业务需求，再列出交互要求

#### AI 产品
- 智能问答意图与 prompt 设计：六列表格（场景、意图分类、意图说明、用户输入示例、prompt 设计、AI 输出示例）
- 评测集及准出标准：明确评测场景、数量、构成及评价维度
- 知识依赖：知识类型、负责人、入库时间

## 输出规则

- 默认输出中文文档。
- 文档命名格式建议：`[编号]PRD_项目名称_V[版本号].md`
- 所有表格使用标准 Markdown 表格语法，表头加粗。
- 颜色、优先级等枚举值保持一致：优先级统一用 P0/P1/P2。
- 避免使用"更直观""更现代""更友好"等空泛词，除非后面紧跟具体定义。
- 无论信息多少，都要产出包含所有必备章节的完整文档，章节内容不足时用"待补充"占位并说明原因。
- 不要在文档中暴露技术实现细节，除非用户明确要求。

## 质量标准

定稿前，检查文档是否满足以下要求：

- 所有必备章节均已包含，无遗漏
- 功能列表中每个功能都有明确的优先级和适用角色
- 非功能需求中的性能指标有具体目标值，不是模糊描述
- 数据埋点覆盖了核心用户行为
- 评审记录中列出了应参与评审的角色
- 所有假设和待确认项都已记录在文档末尾
- 文档可以直接发给评审参与者，无需额外口头解释
