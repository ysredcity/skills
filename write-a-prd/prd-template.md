# PRD文档编写规范要求（Markdown格式）

## 一、文档结构规范
1. **标题层级**
   - 主标题：`#`（一级标题，如"# 03PRD模版_V1.1"）
   - 章节标题：`##`（二级标题，如"## 版本信息"）
   - 子章节标题：`###`（三级标题，如"### a. 用户旅程"）
   - 段落标题：`####`（四级标题，用于细分内容）

2. **必备章节**
   | 章节序号 | 章节名称 | 说明 |
   |----------|----------|------|
   | 一 | 版本信息 | 包含版本号、创建日期、负责人 |
   | 二 | 变更日志 | 记录文档迭代历史 |
   | 三 | 文档说明 | 含名词解释、术语表 |
   | 四 | 需求背景 | 包含需求说明和竞品分析 |
   | 五 | 功能列表 | 按优先级罗列功能点 |
   | 六 | 功能详细说明 | 分数字化/AI产品不同要求 |
   | 七 | 非功能需求 | 性能指标、安全需求等 |
   | 八 | 数据埋点 | 需记录参数名、说明和值 |
   | 附录 | 相关文档 | 附加数据分析/用户调研等报告 |
   | 评审记录 | 评审过程 | 记录评审时间、参与角色和问题 |

## 二、内容编写规范

### 1. 版本信息
- 格式要求：使用表格形式，包含三列（版本号、创建日期、负责人）
- 示例：
  ```markdown
  | 版本号 | 创建日期 | 负责人 |
  |--------|----------|--------|
  | V1.0   | 2025-01-01 | 张三 |
  ```

### 2. 变更日志
- 必须包含四列：时间、版本号、变更人、主要变更内容
- 时间格式：YYYY.MM.DD（如2025.12.08）
- 变更内容需具体明确，如"评审记录增加应参与评审的角色"

### 3. 名词解释
- 采用表格形式，包含"术语/缩略词"和"说明"两列
- 对文档中出现的专业术语、系统简称（如CMS）必须解释
- 示例：
  ```markdown
  | 术语/缩略词 | 说明 |
  |------------|------|
  | CMS | 内容管理系统的简称 |
  ```

### 4. 功能列表
- 必须包含五列：功能名称、功能概述、适用角色、优先级、父记录
- 优先级定义：P0（高）、P1（中）、P2（低）
- 适用角色需明确具体岗位，如"营销中心总产品线经理"

### 5. 功能详细说明
#### 数字化产品必填内容：
- 用户旅程：需使用文档内置画板功能绘制操作路径
- 交互原型图：通过"/Figma"插入设计稿或粘贴内嵌网页链接
- 原型功能说明：需先描述业务需求，再提供交互文档链接

#### AI产品必填内容：
- 智能问答意图与prompt设计：
  - 包含场景、意图分类、意图说明、用户输入示例、prompt设计、AI输出示例六列
  - prompt设计需遵循四原则：简洁明确、上下文记忆、多模态支持、个性化响应
- 评测集及准出标准：明确评测场景、数量、构成及评价维度
- 知识依赖：需与AI知识库产品经理拉通，明确知识类型、负责人及入库时间

### 6. 非功能需求
- 性能指标需包含场景/功能模块、指标名称、定义/计算方法、目标值四列
- 示例：
  ```markdown
  | 场景/功能模块 | 指标名称 | 指标定义/计算方法 | 目标值 |
  |--------------|----------|-------------------|--------|
  | 对话接口 | 平均响应时间 | 请求至接收完整响应的时间平均值 | ≤800ms |
  ```
- 需包含视觉强调、运营配置、法律法规及安全需求说明

## 三、格式规范

### 1. 表格使用
- 统一使用`|`分隔单元格，`---`分隔表头和内容
- 表头文字需加粗（`**表头文字**`）
- 表格内容需左对齐，数字类内容建议右对齐
- 复杂表格（如跨行列）需使用HTML表格语法

### 2. 列表使用
- 无序列表：使用`- `或`* `
- 有序列表：使用`1. `
- 嵌套列表：缩进4个空格
- 示例：
  ```markdown
  - 售前咨询
    - 产品查询
    - 产品对比
  ```

### 3. 链接与图片
- 内部文档链接：`[链接文本](文档URL)`
- 图片插入：`![图片描述](图片URL)`
- 建议图片横向排布以精简篇幅

### 4. 强调文本
- 重要内容：使用`**加粗**`
- 提示性内容：使用`> `引用块
- 示例：
  ```markdown
  > **注意**：需与AI知识库产品经理拉通知识依赖
  ```

## 四、评审规范
1. 评审记录需包含：评审名称、预计时间、应参与角色、主要问题记录、评审状态
2. 应参与评审角色至少包含：
   - 产品经理
   - 产品SE
   - 开发人员（由产品SE指定）
   - TSE、SSE
   - UI设计师（如涉及终端）
   - 用户体验工程师
3. 评审状态包括：未评审、需要二次会议评审、需要二次文档评审、已通过

## 五、其他要求
1. 文档命名格式：`[编号]PRD_项目名称_V[版本号].md`
2. 版本号规则：主版本号.次版本号（如V1.1）
3. 附录需包含数据分析报告、用户调研报告、设计分析报告等支撑材料
4. 涉及用户隐私内容需符合GDPR等相关法规要求
