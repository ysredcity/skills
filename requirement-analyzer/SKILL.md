# Requirement Analyzer

## 概述

将模糊的需求输入（背景、痛点、业务目标、功能想法）分析整理为结构化的需求分析文档。
核心产出三个部分：**产品功能逻辑**、**验收指标**、**需求规模**。

这个 skill 的定位是 PRD 的前置步骤——不是写完整的 PRD，而是把需求"想清楚、说明白"，让后续的评审、排期和 PRD 撰写有一个扎实的起点。

## 工作流程

### 第一步：理解输入，识别关键信息

从用户输入中提取：
- **业务背景**：当前是什么产品/系统，处于什么阶段
- **痛点**：现在的问题是什么，谁在受影响，影响有多大
- **诉求**：用户/业务方希望达成什么目标
- **已有功能逻辑**：用户是否已经描述了功能方案（如果有，作为基础；如果没有，需要推导）
- **参考文档**：用户是否提供了历史 PRD、竞品截图、流程图等（如有，优先参考）

### 第二步：判断是否需要澄清

在以下情况下，**立刻向用户提问**，不要假设后继续：

- **产品特性不明确**：不了解目标产品的核心业务逻辑、用户角色体系、现有功能边界时，无法推导出合理的功能逻辑。此时需要问用户提供产品介绍或相关文档。
- **需求方向存在关键分歧**：同一个痛点有多种截然不同的解法，选择哪个方向会显著影响功能范围和规模。
- **验收标准无法量化**：痛点描述中没有任何可量化的数据基础，无法推导出合理的性能或效果指标。

提问原则：
- 每次最多问 2-3 个问题，聚焦最关键的分歧点
- 问题要具体，不要问"你想要什么效果"这类开放性问题
- 如果可以给出合理假设并继续推进，优先假设，把假设写进文档，不要停下来等待

### 第三步：补充和优化需求

对输入进行分析，确保最终输出满足以下质量标准：

**可量化的价值**
- 痛点有数据支撑（频率、耗时、错误率、影响人数等）
- 目标有可衡量的改善指标（缩短 X%、节省 Y 人时、减少 Z 次点击等）
- 如果用户没有提供数据，根据场景推断合理估算值，并标注为"估算"

**明确的痛点和诉求**
- 痛点描述清楚"谁在什么场景下遇到了什么问题"
- 诉求描述清楚"需要什么能力来解决这个问题"
- 区分"核心诉求"和"延伸诉求"

**产品侧的需求范围边界**
- 明确本期要做什么
- 明确本期不做什么（非本期范围）
- 识别依赖项（需要哪些系统或功能配合）

**可验证的验收标准**
- 每条验收标准都是可观察、可操作的行为描述
- 覆盖正常流程、异常流程和边界情况
- 性能类指标有具体数值

### 第四步：推导产品功能逻辑

这个阶段的功能逻辑是概要级别的，目标是让评审者快速理解"产品层面要做哪几件事"，而不是详细的交互流程或实现方案。

- 用 3-8 条简洁描述覆盖核心能力，每条对应一个独立的产品动作或模块
- 从用户视角出发，写"用户/系统能做什么"，不写"怎么实现"
- 标注每个功能点的优先级（P0/P1/P2）
- 明确本期范围边界：做什么、不做什么、依赖什么
- 如果用户已有功能逻辑，检查是否有遗漏、矛盾或超出范围的内容，简要说明

### 第五步：评估需求规模

根据以下维度综合判断：

| 规模 | 参考标准 |
|------|---------|
| 微需求 | 单一入口或展示调整，无新增流程，1-2 个功能点，不涉及新接口 |
| 小需求 | 局部功能新增或改造，流程清晰，3-5 个功能点，涉及少量新接口 |
| 中需求 | 新功能模块，有完整的用户流程，6-15 个功能点，涉及多个接口或系统联动 |
| 大需求 | 跨系统或跨业务线，流程复杂，15 个以上功能点，涉及架构或数据模型变更 |

评估时说明判断依据，不要只给结论。

## 输出格式

使用 `references/output-template.md` 作为文档结构模板，包含五个章节：需求背景与价值、产品功能逻辑、验收指标、需求规模、待确认问题 / 默认假设。

## 输出规则

- 默认输出中文文档。
- 优先使用用户原有术语，保持业务语义一致。
- 功能描述写"用户可以做什么"，不写技术实现方式。
- 验收标准每条独立成行，格式统一。
- 规模评估必须附判断依据，不能只给结论。
- 如果某个章节信息不足，用合理假设填充并标注"[假设]"，不要留空或写"待补充"。
- 避免使用"更好用""更高效""更直观"等空泛词，除非后面紧跟具体定义。
