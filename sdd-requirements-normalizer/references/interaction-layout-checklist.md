# 交互与布局补全清单

当原始需求描述比较模糊，尤其是面向原型产出时，优先使用这份清单补齐信息。

## 交互
- 用户是通过什么触发进入这个页面或流程的？
- 页面上的主要操作是什么？
- 次要操作有哪些？
- 每个关键操作完成后，页面或系统应如何反馈？
- 加载、成功、失败、校验不通过、危险操作时分别要展示什么反馈？
- 哪些操作需要二次确认？
- 哪些内容适合页内直接编辑，哪些适合弹窗、抽屉或独立页面？
- 是否存在键盘操作、触控操作、手势操作等特殊要求？

## 布局
- 页面由哪些主要区块组成？
- 哪些信息必须默认可见，不能埋得太深？
- 哪些内容应该放在最上方或最显著位置？
- 适合使用卡片、表格、列表、标签页、分栏、渐进展开中的哪种结构？
- 搜索、筛选、排序、批量操作应该放在哪里？
- 哪些内容需要固定、折叠、滚动、分页或悬浮？
- 在更小尺寸的屏幕下，布局应该如何变化？

## 状态
- 无数据时用户看到什么？
- 数据加载中时用户看到什么？
- 数据不完整时用户看到什么？
- 因权限、额度、校验问题导致操作受阻时用户看到什么？

## 原型优先原则
- 优先写清楚页面区块和结构，不要只写抽象体验词。
- 优先写清楚状态切换和反馈，不要只写“支持交互”。
- 优先写清楚信息层级，不要只写“布局清晰”。
- 优先用合理假设推动原型落地，而不是因为信息不完备就停住。
