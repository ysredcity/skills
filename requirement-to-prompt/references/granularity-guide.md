## 概览
需求颗粒度识别是决定如何处理产品需求的关键步骤。准确识别需求的详细程度，可以避免过度补充或遗漏关键信息。

## 详细需求特征

### 判断标准
详细需求通常包含以下特征中的3个及以上：

1. **目标描述完整**：
   - 包含明确的用户场景
   - 描述了核心价值
   - 说明了解决的问题

2. **设计风格明确**：
   - 指定了主色调或色系
   - 说明了字体规范
   - 提到了组件风格（如 shadcn/ui、Ant Design）
   - 描述了整体视觉风格（企业风、极简风等）

3. **页面结构清晰**：
   - 明确列出了页面区块（页头、内容区、操作区等）
   - 详细描述了每个区块的字段
   - 说明了交互规则（只读、可编辑、弹窗等）
   - 包含响应式适配说明（PC端/移动端差异）

4. **实现要求具体**：
   - 指定了技术栈（Tailwind、React、Vue等）
   - 说明了文件组织方式
   - 描述了数据处理方式（mock数据、API接口）
   - 包含交互细节要求（加载动画、过渡效果）

### 处理策略
- **直接使用**：保留需求中的所有描述
- **格式化输出**：确保符合标准模板的四个部分
- **不做补充**：避免引入与原需求冲突的内容

## 简略需求特征

### 判断标准
简略需求通常表现为以下形式：

1. **一句话描述**：
   - 示例："做一个用户管理页面"
   - 示例："需要一个商品列表"

2. **功能点罗列**：
   - 示例："需要列表页、搜索、筛选、分页"
   - 示例："支持新增、编辑、删除用户"

3. **业务场景描述**：
   - 示例："客服需要查看和回复用户反馈"
   - 示例："运营人员需要管理商品信息"

### 处理策略
- **智能补充**：基于页面类型补充标准模板
- **合理假设**：采用行业最佳实践作为默认值
- **标注说明**：在补充内容后标注"可根据实际情况调整"

## 处理策略对比

| 维度 | 详细需求 | 简略需求 |
|------|---------|---------|
| 目标描述 | 保留原描述 | 基于页面类型生成标准目标 |
| 设计风格 | 保留原描述 | 根据目标用户补充默认风格 |
| 页面结构 | 保留原描述 | 根据页面类型补充标准区块 |
| 实现要求 | 保留原描述 | 补充响应式和技术栈规范 |
| 补充范围 | 不补充 | 智能补充所有缺失部分 |
| 输出标注 | 无需标注 | 标注"可根据实际情况调整" |

## 边界情况处理

### 部分详细的需求
**特征**：某些部分详细，某些部分简略

**处理策略**：
- 详细部分：保留原描述
- 简略部分：按简略需求策略补充
- 示例：有详细的页面结构，但没有设计风格 → 保留结构，补充设计风格

### 模糊的需求
**特征**：描述不清晰，存在歧义

**处理策略**：
- 采用最小可用方案
- 在补充内容中标注为"建议实现"
- 提供可扩展的架构说明

### 冲突的需求
**特征**：不同部分的描述存在矛盾

**处理策略**：
- 优先级：明确描述 > 隐含描述 > 默认模板
- 在输出中标注冲突点
- 提供多种解决方案选项

## 实例分析

### 实例1：详细需求
**输入**：
```
目标：生成一个商品管理列表页，面向B端用户，支持PC与移动端响应式适配。

设计风格：B端企业风格，主色调蓝色，使用 Ant Design 组件

页面结构：
1. 搜索筛选区：商品名称、分类、状态筛选
2. 列表数据区：表格展示，支持排序
3. 操作区：新增、编辑、删除按钮

实现要求：使用 React + Ant Design，支持分页和批量操作
```

**识别**：详细需求
**处理**：直接格式化输出，不做补充

### 实例2：简略需求
**输入**："需要一个用户管理页面，B端用户"

**识别**：简略需求
**处理**：补充完整的目标、设计风格、页面结构、实现要求

**补充结果**：
```
目标：生成一个企业级用户管理页面，面向B端管理员，支持PC与移动端响应式适配

设计风格：
- 整体采用 B端企业管理系统风格：克制、专业
- 主色调使用品牌色#00aaa6
- 字体选用 PingFang SC / HarmonyOS Sans
- 组件参考企业级设计规范

页面结构：
1. 搜索筛选区：用户名、角色、状态筛选
2. 列表数据区：表格展示，支持排序和分页
3. 操作区：新增、编辑、删除按钮

实现要求：
- 响应式适配：支持PC、移动端正常显示
- 技术实现：单文件 HTML + Tailwind CDN + 原生 JS（可根据实际情况调整）
- 交互逻辑：包含表单验证、删除确认、操作反馈

（注：以上为基于B端用户管理场景的默认补充，可根据实际情况调整）
```
